# A2. Component-Decision Table

Below is the component-decision table for the new GeekShop observability stack. All chosen components are consolidated within the Grafana ecosystem to maximize integration capability and minimize licensing costs.

| Capability | Chosen Component / Vendor | Why this one | What gets worse if we change our mind in six months |
| :--- | :--- | :--- | :--- |
| **Metrics Ingestion + Storage + Query** | **Grafana Cloud Metrics (Mimir)** | It is a fully managed, horizontally scalable, Prometheus-compatible backend that natively supports Exemplars, enabling 1-click transitions from metrics spikes to traces. | We would have to rewrite all PromQL-based dashboards, recreate scraping rules, and risk losing historical metrics data during a complex migration of millions of active time-series database records. |
| **Logs Ingestion + Storage + Search** | **Grafana Cloud Logs (Loki)** | Loki's metadata-only indexing model reduces log storage and ingestion costs by over 90% compared to Splunk while natively linking log lines to traces using Trace ID. | Migrating terabytes of compressed raw logs from S3 back to a full-text indexing system (like Splunk or OpenSearch) would be extremely expensive, and we would lose the native Log-to-Trace click-through correlation. |
| **Distributed Tracing** | **Grafana Cloud Traces (Tempo)** | Tempo provides high-scale trace storage backed by cheap S3 object storage, enabling us to implement tail-based sampling to capture 100% of errors/slow requests at minimal cost. | We would have to re-configure OpenTelemetry SDK exporters on all 10 microservices and lose the automatic Exemplar-based transition from metrics to traces. |
| **Alerting (Rule Engine) + Correlation / Grouping** | **Grafana Alerting** | It centralizes rule evaluation directly against Mimir and Loki, consolidating all alerting logic in a single interface instead of maintaining scattered rules in Datadog and Splunk. | We would have to recreate and re-test hundreds of complex alerting rules across multiple different vendor dashboards, increasing the risk of missing critical outages. |
| **Incident Routing + Paging** | **Grafana OnCall** | It is natively integrated with Grafana Alerting, provides automated alert grouping by fingerprint, and manages on-call schedules at 75% lower cost than PagerDuty. | We would have to rebuild all on-call rotations, escalation policies, and integration webhooks in a separate system (like PagerDuty or Opsgenie), creating significant operational friction. |
| **Dashboards + SLO Tracking** | **Grafana Cloud (Unified Console)** | It acts as the single pane of glass for all signals, removing the cognitive load of developers context-switching across 4 different vendor UIs during a production incident. | We would have to rebuild all executive and engineering dashboards from scratch across disjointed vendor interfaces, and developers would regress to context-switching during incident triage. |
