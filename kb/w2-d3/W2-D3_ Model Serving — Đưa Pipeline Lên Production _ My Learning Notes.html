Model Serving — Đưa Pipeline Lên Production#

D1 + D2 cho bạn 2 layer chạy được trong notebook. Notebook không phải production. Hôm nay biến nó thành API service mà 1 hệ thống monitoring có thể POST alert vào và nhận lại incident report.

Khi nói “serving” trong AIOps, không chỉ về serve ML model — nó là về serve toàn bộ pipeline (correlation + RCA + LLM call) như 1 unit có:

  • HTTP endpoint
  • Latency budget
  • Health check
  • Versioning + rollback
  • Monitoring của chính bản thân nó

Cuối ngày bạn có serve.py chạy được, đó là Layer 4 của lab Friday.

Rule of thumb: Pipeline trong notebook khác với pipeline production ở 3 thứ — concurrency, failure handling, và observability. Đừng đợi production để nghĩ về 3 thứ này.

1. Framework — FastAPI vs Flask vs BentoML#

3 framework phổ biến cho Python serving. Tóm gọn:

FrameworkKhi nào dùngƯuNhược
FlaskQuick prototype, họcSimple, ít magicSync only (cần workaround cho async), không validate input native
FastAPIProduction API, mixed workloadAsync, Pydantic validation, OpenAPI auto, type hintsMagic hơn Flask một chút
BentoMLML model–centricModel versioning, batching native, Yatai deployHọc curve, overhead cho non-ML workload

Cho lab W2 dùng FastAPI. Lý do: pipeline có LLM call (IO bound, hưởng từ async), input có schema (Pydantic dễ), trainer test bằng curl/requests (OpenAPI giúp document).

uv pip install fastapi uvicorn pydantic

2. Endpoint cơ bản#

2.1 Skeleton#

# serve.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field
from typing import Optional
import logging

logging.basicConfig(level=logging.INFO, format='%(asctime)s %(levelname)s %(message)s')
logger = logging.getLogger('aiops')

app = FastAPI(
    title='AIOps Incident Pipeline',
    version='1.0.0',
    description='Group lab W2 — correlate alerts → RCA → suggest action',
)

# Pydantic schema — input validation
class Alert(BaseModel):
    id: str
    ts: str
    service: str
    metric: str
    severity: str
    value: float
    threshold: float
    labels: Optional[dict] = Field(default_factory=dict)

class IncidentRequest(BaseModel):
    alerts: list[Alert]

# Output schema (match W2_lab_spec.md Section 4)
class Cluster(BaseModel):
    cluster_id: str
    alert_count: int
    services: list[str]
    time_range: list[str]

class RootCause(BaseModel):
    service: str
    confidence: float
    reasoning: str

class SimilarIncident(BaseModel):
    id: str
    similarity: float
    summary: str

class IncidentResponse(BaseModel):
    clusters: list[Cluster]
    root_cause: RootCause
    recommended_actions: list[str]
    similar_incidents: list[SimilarIncident]


@app.get('/healthz')
def healthz() -> dict:
    return {'status': 'ok'}


@app.post('/incident', response_model=IncidentResponse)
def post_incident(req: IncidentRequest) -> IncidentResponse:
    """
    Process batch of alerts → return incident report.
    
    Chain: correlate (L1) → rca (L2) → enrich w/ LLM (L3) → assemble response.
    """
    logger.info(f"Received {len(req.alerts)} alerts")
    
    if not req.alerts:
        raise HTTPException(status_code=400, detail='Empty alert list')
    
    # ... pipeline calls go here ...
    # Stub:
    return IncidentResponse(
        clusters=[],
        root_cause=RootCause(service='unknown', confidence=0.0, reasoning='not implemented'),
        recommended_actions=[],
        similar_incidents=[],
    )

2.2 Pydantic validation — không phải optional#

Nếu input sai format (e.g. missing service field), Pydantic auto trả 422 với detail. Không cần code thêm.

$ curl -X POST http://localhost:8000/incident -H "Content-Type: application/json" \
       -d '{"alerts":[{"id":"a-1"}]}'

# 422 Unprocessable Entity
# {"detail":[{"loc":["body","alerts",0,"ts"],"msg":"field required","type":"value_error.missing"}, ...]}

Trainer test sẽ kiểm tra cả happy path lẫn malformed input. Đảm bảo endpoint không trả 500 khi input sai.

3. Chain 3 Layer Lại#

Pipeline assembly — gọi correlate.py → rca.py → enrich.py:

# pipeline.py — glue layer
from correlate import correlate, build_graph_from_json
from rca import run_rca
from enrich import call_llm_action_suggest  # team L3 code
import json
from pathlib import Path

# Load once at module level (cached)
GRAPH = build_graph_from_json('lab/dataset/services.json')
HISTORY = json.loads(Path('lab/dataset/incidents_history.json').read_text())['incidents']


def process_batch(alerts: list[dict]) -> dict:
    """
    Full pipeline. Trả về dict matching IncidentResponse schema.
    """
    # L1: Correlate
    clusters = correlate(alerts, GRAPH, gap_sec=120, max_hop=2)
    
    if not clusters:
        return {
            'clusters': [],
            'root_cause': {'service': 'unknown', 'confidence': 0, 'reasoning': 'No clusters'},
            'recommended_actions': [],
            'similar_incidents': [],
        }
    
    # Pick largest cluster as "primary incident"
    primary = max(clusters, key=lambda c: c['alert_count'])
    
    # L2 + L3: RCA + LLM enrichment cho primary cluster
    rca_result = run_rca(primary, alerts, GRAPH, HISTORY)
    
    # Assemble response
    return {
        'clusters': [
            {
                'cluster_id': c['cluster_id'],
                'alert_count': c['alert_count'],
                'services': c['services'],
                'time_range': c['time_range'],
            }
            for c in clusters
        ],
        'root_cause': {
            'service': rca_result['root_cause'],
            'confidence': rca_result['confidence'],
            'reasoning': rca_result.get('reasoning', ''),
        },
        'recommended_actions': rca_result.get('actions', []),
        'similar_incidents': [
            {'id': inc_id, 'similarity': 0.7, 'summary': '...'}  # populate từ history
            for inc_id in rca_result.get('similar_incidents', [])[:3]
        ],
    }

Sau đó cập nhật serve.py:

from pipeline import process_batch

@app.post('/incident', response_model=IncidentResponse)
def post_incident(req: IncidentRequest) -> IncidentResponse:
    alerts_dict = [a.model_dump() for a in req.alerts]
    try:
        result = process_batch(alerts_dict)
    except Exception as e:
        logger.error(f"Pipeline failed: {e}", exc_info=True)
        raise HTTPException(status_code=500, detail=f'Pipeline error: {e}')
    return IncidentResponse(**result)

4. Latency Budget#

4.1 Đo trước, optimize sau#

Add middleware đo latency mỗi request:

import time
from fastapi import Request

@app.middleware('http')
async def add_timing(request: Request, call_next):
    start = time.perf_counter()
    response = await call_next(request)
    duration_ms = (time.perf_counter() - start) * 1000
    response.headers['X-Response-Time-Ms'] = f'{duration_ms:.1f}'
    logger.info(f"{request.method} {request.url.path} {response.status_code} {duration_ms:.0f}ms")
    return response

Trên scenario chính (200 alert), latency breakdown thường:

PhaseTime%
Pydantic parse + validate5-10ms0.2%
Layer 1 correlate (Python loops)50-100ms2%
Layer 2 graph RCA20-50ms1%
Layer 3 LLM call (network)3-8s95-97%
Response serialize5-10ms0.2%
Total3.5-8.5s

LLM call dominate. Tối ưu 100ms ở L1/L2 = 1% improvement. Tối ưu LLM = lớn.

4.2 Optimization cho LLM call#

  1. Cache — same prompt → cached response (hash prompt):

    from functools import lru_cache
import hashlib

_llm_cache: dict[str, dict] = {}

def cached_llm_call(prompt: str) -> dict:
    key = hashlib.sha256(prompt.encode()).hexdigest()
    if key not in _llm_cache:
        _llm_cache[key] = _do_llm_call(prompt)
    return _llm_cache[key]

    Cảnh báo: cache không clear → memory grow. Dùng cachetools.TTLCache(maxsize=1000, ttl=3600).

  2. Async + concurrent — nếu cần gọi LLM cho 3 cluster, gọi 3 song song thay vì sequential:

    import asyncio

async def llm_call_async(prompt: str) -> dict:
    # Use openai async client
    ...

@app.post('/incident')
async def post_incident(req: IncidentRequest):
    ...
    tasks = [llm_call_async(p) for p in prompts]
    results = await asyncio.gather(*tasks)

  3. Smaller model — gpt-4o-mini thay gpt-4o, hoặc gemini-flash. 5x cheap + 2x faster.

  4. Skip LLM — nếu graph RCA confidence ≥ 0.9, không cần LLM enrichment.

4.3 Timeout#

Set timeout cho mọi outbound call:

from openai import OpenAI
client = OpenAI(timeout=10.0, max_retries=2)  # 10s timeout, 2 retry

# Hoặc với requests:
# requests.post(url, json=..., timeout=(3, 10))  # connect=3s, read=10s

Không timeout → 1 LLM call hang → endpoint hang forever → trainer demo fail.

5. Concurrency#

5.1 1 worker vs N worker#

Default uvicorn serve:app chạy 1 worker. 1 worker = 1 process xử lý request sequential (cho sync code) hoặc concurrent với event loop (cho async code).

Để scale cho production, dùng nhiều worker:

uvicorn serve:app --host 0.0.0.0 --port 8000 --workers 4

Khi 4 worker, request được load balance round-robin. Trade-off: nhiều worker → nhiều memory (process duplicate state).

5.2 Race condition với shared state#

Nếu pipeline có in-memory cache (e.g. dedup fingerprint store) + nhiều worker → mỗi worker có copy riêng → dedup không cross-worker.

Solution: stateless — mỗi request load state từ Redis hoặc DB. Hoặc giữ stateless trong lab (chấp nhận limitation, document trong reflection).

Lab này chấp nhận stateless / single-worker — không phải production scale exercise. Quan trọng là biết trade-off, không phải implement perfect.

5.3 Concurrent test#

Test endpoint chịu được concurrent request:

# Apache Bench: 100 request, 10 concurrent
ab -n 100 -c 10 -p alerts.json -T application/json http://localhost:8000/incident

# wrk: 10 thread, 30s
wrk -t10 -c50 -d30s --script post.lua http://localhost:8000/incident

Theo dõi p50 / p99 + error rate. Nếu p99 > 30s hoặc error > 1% → cần fix.

6. Health Check + Readiness#

/healthz (section 2) đủ cho liveness. Cho readiness (k8s rolling deploy), thêm:

@app.get('/readyz')
def readyz() -> dict:
    """Check downstream dependencies. Trả 503 nếu chưa ready."""
    checks = {}
    
    # Check graph loaded
    checks['graph'] = GRAPH.number_of_nodes() > 0
    
    # Check history loaded
    checks['history'] = len(HISTORY) > 0
    
    # Check LLM API (optional, có thể skip để readiness không depend external)
    try:
        # quick model list call
        from openai import OpenAI
        OpenAI(timeout=2.0).models.list()
        checks['llm'] = True
    except Exception:
        checks['llm'] = False
    
    if not all(checks.values()):
        raise HTTPException(status_code=503, detail=checks)
    
    return {'status': 'ready', 'checks': checks}

Lý do tách /healthz vs /readyz:

  • healthz = process còn sống (k8s restart pod nếu fail). Không depend external.
  • readyz = sẵn sàng nhận traffic (k8s remove from load balancer nếu fail). Depend external.

7.2 Feature flag cho LLM#

import os
USE_LLM = os.environ.get('AIOPS_USE_LLM', 'true').lower() == 'true'

def run_rca_with_flag(cluster, alerts, graph, history):
    if not USE_LLM:
        # Skip LLM, return graph-only result
        candidates = rca_combined(cluster, alerts, graph)
        return {
            'root_cause': candidates[0][0],
            'confidence': candidates[0][1],
            'method': 'graph-only-flag-off',
            ...
        }
    return run_rca(cluster, alerts, graph, history)

Khi LLM provider outage, set AIOPS_USE_LLM=false + restart → endpoint vẫn chạy với graph only.

7.3 Shadow deployment (concept)#

Production AIOps: deploy v2 cùng v1, nhưng v2 không serve traffic — nó shadow v1 (nhận cùng input, log output). So sánh v2 output với v1 output trên data thật. Nếu OK trong 1 tuần → promote v2.

Không implement trong lab (out of scope). Đề cập để biết direction. Trong vấn đáp, biết khái niệm → +1 điểm Layer 4.

8. Self-Monitoring#

Pipeline AIOps monitor production — nhưng bản thân nó cũng cần monitor.

8.1 Metric Prometheus#

from prometheus_client import Counter, Histogram, make_asgi_app

# Metrics
REQUEST_COUNT = Counter('aiops_incident_requests_total', 'Total incident requests', ['status'])
REQUEST_LATENCY = Histogram('aiops_incident_latency_seconds', 'Incident pipeline latency')
LLM_FAILURES = Counter('aiops_llm_failures_total', 'LLM call failures', ['reason'])
CLUSTER_COUNT = Histogram('aiops_clusters_per_request', 'Clusters produced per request')

# Mount Prometheus app on /metrics
app.mount('/metrics', make_asgi_app())

# Use in endpoint
@app.post('/incident')
async def post_incident(req: IncidentRequest):
    with REQUEST_LATENCY.time():
        try:
            result = process_batch([a.model_dump() for a in req.alerts])
            REQUEST_COUNT.labels(status='success').inc()
            CLUSTER_COUNT.observe(len(result['clusters']))
            return IncidentResponse(**result)
        except Exception as e:
            REQUEST_COUNT.labels(status='error').inc()
            raise

8.3 Logging#

Structured log (JSON), không print:

import logging
import json

class JsonFormatter(logging.Formatter):
    def format(self, record):
        log_obj = {
            'ts': self.formatTime(record),
            'level': record.levelname,
            'msg': record.getMessage(),
            'logger': record.name,
        }
        if hasattr(record, 'extra'):
            log_obj.update(record.extra)
        return json.dumps(log_obj)

handler = logging.StreamHandler()
handler.setFormatter(JsonFormatter())
logger = logging.getLogger('aiops')
logger.addHandler(handler)
logger.setLevel(logging.INFO)

# Use:
logger.info('Processed incident', extra={'extra': {
    'cluster_count': 3,
    'root_cause': 'payment-svc',
    'confidence': 0.84,
}})

JSON log dễ ship vào ELK/Loki, query bằng cluster_count > 5 thay vì grep.

9.1 Unit test — pure function#

# tests/test_correlate.py
import pytest
from correlate import fingerprint, session_groups

def test_fingerprint_excludes_timestamp():
    a = {'service': 'payment-svc', 'metric': 'latency', 'severity': 'crit', 'ts': '2026-06-12T09:42:01Z', 'value': 1840}
    b = {'service': 'payment-svc', 'metric': 'latency', 'severity': 'crit', 'ts': '2026-06-12T09:42:30Z', 'value': 1900}
    assert fingerprint(a) == fingerprint(b)

def test_session_split_on_gap():
    alerts = [
        {'id': '1', 'ts': '2026-06-12T09:42:00Z', 'service': 'a', 'metric': 'm', 'severity': 'warn', 'value': 1, 'threshold': 0},
        {'id': '2', 'ts': '2026-06-12T09:42:30Z', 'service': 'a', 'metric': 'm', 'severity': 'warn', 'value': 1, 'threshold': 0},
        {'id': '3', 'ts': '2026-06-12T09:50:00Z', 'service': 'a', 'metric': 'm', 'severity': 'warn', 'value': 1, 'threshold': 0},
    ]
    groups = session_groups(alerts, gap_sec=120)
    assert len(groups) == 2  # 1 + 2 split bởi 7.5 phút gap

11. Embedded Exercise#

Task: Code serve.py + DESIGN.md cho lab.

Quy ước nộp bài:

  • Branch main
  • Path: aiops-<tên>/w2/d3/ (lowercase w2 + d3)
  • File: serve.py + DESIGN.md (HOA) + SUBMIT.md (HOA)
  • Tên file chính xác như trên — không phải app.py, architecture.md, hoặc adr_001.md. Sai naming = 1 điểm.

11.1 Steps#

  1. Tạo aiops-<your-name>/w2/d3/
  2. Build skeleton serve.py theo section 2-3
  3. Add health/ready endpoints (section 6)
  4. Add latency middleware (section 4.1)
  5. Tích hợp với D1 correlate + D2 RCA (giả lập tạm — không cần chính xác, chỉ cần wire)
  6. Write DESIGN.md ≥ 100 từ:
    • Pipeline architecture trong endpoint
    • Latency budget breakdown
    • 1 production concern (concurrency hoặc fault tolerance) — bạn handle thế nào
    • Trade-off: vì sao chọn FastAPI thay vì Flask/BentoML
  7. Write SUBMIT.md với reflection

11.2 Acceptance#

  • serve.py chạy với uvicorn serve:app --port 8000

12. EOD Checkpoint#

Trong SUBMIT.md:

  1. Latency budget của endpoint bạn là bao nhiêu (p99)? Phase nào chiếm thời gian nhất?
  2. Endpoint của bạn xử lý 1 input với 5 alert vs 500 alert — sự khác biệt latency là gì? Linear scale? Hay có phần fixed cost?
  3. LLM provider down giữa demo. Hệ thống bạn behave thế nào? Phương án dự phòng?
  4. /healthz và /readyz khác nhau gì? Khi nào dùng cái nào?
  5. Trainer POST 4 request đồng thời từ 4 nhóm khác nhau. Endpoint bạn handle ổn không? Bottleneck đầu tiên?
