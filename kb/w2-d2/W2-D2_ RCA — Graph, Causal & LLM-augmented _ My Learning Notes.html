RCA — Graph, Causal & LLM-augmented#

Hôm qua bạn đi từ 200 alert → 3-7 cluster. Mỗi cluster bây giờ là 1 “đám cháy” với 5-15 service liên quan. Câu hỏi tiếp theo: trong cluster này, service nào là ngọn lửa gốc, service nào là khói?

Đây là Root Cause Analysis (RCA). Ngày hôm nay cover 3 cách làm RCA tự động:

  1. Graph traversal — dùng service graph để pick “node central nhất” trong cluster
  2. Causal inference — dùng metric time-series để xác định service nào degrade trước
  3. LLM-augmented — hỏi LLM với context (history + graph) để suggest root cause

Cả 3 không loại trừ nhau — production system tốt combine cả 3 và return ranking.

Rule of thumb: Đừng kỳ vọng RCA tự động đạt accuracy 100%. Target thực tế: top-3 candidate chứa root cause đúng > 80% các case. SRE sẽ verify.

1. Vấn Đề: Victim vs Culprit#

Khi payment-svc, checkout-svc, edge-lb đều alert cùng lúc, intuition đầu tiên là “service nào có nhiều alert nhất → root cause”. Sai.

Trong scenario chính của lab:

  • payment-svc — 14 alert (gây)
  • checkout-svc — 12 alert (victim — depend on payment)
  • edge-lb — 9 alert (victim — depend on checkout)
  • notification-svc — 6 alert (victim — kafka backlog do checkout chậm)

checkout-svc không phải root cause. Nó alert nhiều vì mọi user request đi qua nó, và khi payment hỏng, checkout là chỗ đầu tiên thấy lỗi.

Phân biệt culprit (kẻ gây) vs victim (nạn nhân) là toàn bộ challenge của RCA. Có 3 dấu hiệu giúp phân biệt:

Dấu hiệuCulprit cóVictim có
TimestampAlert sớm nhấtAlert muộn (sau cascade lan tới)
Topology positionDownstream — service-as-dependencyUpstream — service-as-caller
IndependenceAlert dù mọi thứ khác bình thườngAlert chỉ khi service khác hỏng

3 algorithm sau lần lượt khai thác 3 dấu hiệu này.

2. Graph Traversal RCA#

2.1 Insight#

Trong service graph A → B nghĩa là A depend on B. Nếu B hỏng, A cũng hỏng. Tức là hỏng propagate ngược chiều edge (từ dependency lên dependent).

Vậy nếu cluster có nhiều service alert, ai gần “đáy” (downstream-most) thường là root cause.

2.2 Simple algorithm — pick downstream-most#

import networkx as nx

def downstream_most(cluster_services: list[str], graph: nx.DiGraph) -> str:
    """
    Trả về service ở 'đáy' nhất của subgraph cluster.
    
    Subgraph: chỉ giữ service trong cluster_services.
    'Downstream-most' = node có out-degree thấp nhất trong subgraph 
    (không gọi ai, chỉ bị gọi).
    
    Khi tie, pick service có in-degree cao nhất (bị nhiều service phụ thuộc nhất).
    """
    sub = graph.subgraph(cluster_services).copy()
    
    # Score = (in_degree, -out_degree) — cả 2 cao là sign of root cause
    scores = []
    for node in sub.nodes():
        scores.append((node, sub.in_degree(node), -sub.out_degree(node)))
    
    scores.sort(key=lambda x: (x[2], -x[1]))  # nhỏ out_degree trước, lớn in_degree trước
    return scores[0][0]

Mental test: Cluster = [edge-lb, checkout-svc, payment-svc].

  • edge-lb: in=0, out=1 (calls checkout)
  • checkout-svc: in=1, out=1 (calls payment, called by edge-lb)
  • payment-svc: in=1, out=0 (terminal — không call ai trong cluster)

→ payment-svc selected. ✔ Match scenario.

2.3 PageRank scoring#

Pick top-1 deterministic không robust. Khi cluster có 8 service và 2 candidate sát nhau, ta cần score liên tục thay vì binary.

PageRank intuition: node được “vote” bởi các node link tới nó. Trong RCA, ta dùng PageRank trên reverse graph — node có nhiều caller depend → score cao.

def rca_pagerank(cluster_services: list[str], graph: nx.DiGraph, alpha: float = 0.85) -> dict[str, float]:
    """
    PageRank trên reverse subgraph. Trả về dict service → score.
    
    Reverse graph: nếu A → B trong service graph (A calls B), 
    thì trong reverse graph B → A. PageRank trên reverse = đo "B được dùng nhiều thế nào".
    
    Why alpha=0.85: damping factor mặc định Google PageRank. Có thể giảm 0.5 nếu 
    muốn node start (alert đầu) có weight cao hơn.
    """
    sub = graph.subgraph(cluster_services).copy()
    reverse = sub.reverse(copy=True)
    
    if len(reverse) == 0:
        return {}
    
    scores = nx.pagerank(reverse, alpha=alpha)
    return scores

Output cho cluster scenario chính:

payment-svc:    0.42  ← top
checkout-svc:   0.28
edge-lb:        0.15
notification-svc: 0.08
cart-svc:       0.04
inventory-svc:  0.03

Confidence: score_top / sum(scores) → 0.42 → “moderately confident”.

2.4 Kết hợp với timestamp#

Graph alone không phân biệt “service A đứng dưới graph nhưng alert sau service B đứng trên”. Bổ sung temporal: service alert sớm nhất trong cluster có boost.

def rca_combined(cluster: dict, alerts: list[dict], graph: nx.DiGraph) -> list[tuple[str, float]]:
    """
    Combine PageRank + earliest-alert timestamp.
    
    Cho mỗi service trong cluster:
      final_score = 0.6 * pagerank_normalized + 0.4 * timestamp_score
    
    Timestamp score: 1.0 cho service alert sớm nhất, 0.0 cho muộn nhất, linear.
    """
    cluster_services = cluster['services']
    pr_scores = rca_pagerank(cluster_services, graph)
    
    # Earliest alert mỗi service
    earliest_by_svc = {}
    for a in alerts:
        if a['service'] in cluster_services:
            ts = a['ts']
            if a['service'] not in earliest_by_svc or ts < earliest_by_svc[a['service']]:
                earliest_by_svc[a['service']] = ts
    
    if not earliest_by_svc:
        return [(s, pr_scores.get(s, 0)) for s in cluster_services]
    
    sorted_by_time = sorted(earliest_by_svc.items(), key=lambda x: x[1])
    n = len(sorted_by_time)
    
    timestamp_score = {svc: 1.0 - i / max(n - 1, 1) for i, (svc, _) in enumerate(sorted_by_time)}
    
    # Normalize pagerank
    pr_max = max(pr_scores.values()) if pr_scores else 1.0
    
    combined = []
    for svc in cluster_services:
        pr = pr_scores.get(svc, 0) / pr_max if pr_max > 0 else 0
        ts = timestamp_score.get(svc, 0)
        score = 0.6 * pr + 0.4 * ts
        combined.append((svc, score))
    
    combined.sort(key=lambda x: -x[1])
    return combined

2.5 Edge case — terminal noise#

Node “downstream-most” có thể là database hoặc cache. Nếu DB alert vì pool full do app leak connection, DB là victim không phải culprit.

Giải pháp:

  • Tag node với criticality + type trong service graph (services.json đã có)
  • Nếu top-1 là store (DB / cache), check xem nó alert trước hay sau application service depend trên nó
  • Nếu sau → app là culprit, DB là victim
  • Nếu trước → DB là culprit thật (e.g. DB crash, network partition)

3. Causal Inference RCA#

Graph traversal không cần data từ metric. Nó chỉ dùng topology + alert. Causal inference ngược lại — chỉ cần metric time-series, không cần graph.

3.1 Granger causality — direction theo time#

Câu hỏi: “Metric A của payment-svc có ‘cause’ metric B của checkout-svc không?”

Granger test trả lời: Nếu giá trị quá khứ của A giúp dự đoán B tốt hơn so với chỉ dùng quá khứ của B, ta nói A Granger-cause B.

Đây không phải causation theo nghĩa philosophy — nó là “temporal precedence + predictive”. Nhưng trong AIOps đủ tốt cho purposes.

import numpy as np
from statsmodels.tsa.stattools import grangercausalitytests

def granger_pair(series_a: np.ndarray, series_b: np.ndarray, maxlag: int = 5) -> float:
    """
    Test "A Granger-causes B" — return p-value của test.
    p < 0.05 → reject null → A có vẻ Granger-cause B.
    
    Args:
        series_a, series_b: numpy 1D arrays cùng độ dài, stationary
        maxlag: max số lag để test
    
    Returns:
        min p-value across lags
    """
    data = np.column_stack([series_b, series_a])  # statsmodels nhận (Y, X)
    try:
        result = grangercausalitytests(data, maxlag=maxlag, verbose=False)
        p_values = [result[lag][0]['ssr_ftest'][1] for lag in range(1, maxlag + 1)]
        return min(p_values)
    except Exception:
        return 1.0  # test fail → no evidence

3.2 Cảnh báo về Granger#

  1. Yêu cầu stationary: series không có trend / seasonality. Phải np.diff() trước, hoặc dùng STL decompose (W1-D1 đã học).
  2. Chỉ pairwise: test A→B, không phải A→B trong context có C. Có thể nói “A causes B” trong khi thực tế cả 2 đều do C cause.
  3. Sample size: cần ≥ 50 data points, lý tưởng 200+. 5 phút × scrape 15s = 20 điểm — không đủ. Phải dùng raw metric, không chỉ alert.

Với constraint sample size, Granger không phải first-line tool cho RCA. Nó hữu ích khi bạn có sẵn time-series dài + muốn build causal graph offline.

3.3 Cross-correlation lag — đơn giản hơn#

Một cách approximation: chỉ đo lag mà 2 metric tương quan max.

def lag_correlation(series_a: np.ndarray, series_b: np.ndarray, max_lag: int = 30) -> tuple[int, float]:
    """
    Tìm lag k mà corr(A[t], B[t+k]) max.
    
    k > 0 → A đi trước B (A có thể cause B)
    k < 0 → B đi trước A (B có thể cause A)
    k = 0 → cùng lúc → không kết luận direction
    
    Returns:
        (best_lag, best_correlation)
    """
    n = len(series_a)
    best = (0, -2.0)
    for k in range(-max_lag, max_lag + 1):
        if k > 0:
            a, b = series_a[:-k], series_b[k:]
        elif k < 0:
            a, b = series_a[-k:], series_b[:k]
        else:
            a, b = series_a, series_b
        
        if len(a) < 10:
            continue
        
        corr = np.corrcoef(a, b)[0, 1]
        if abs(corr) > abs(best[1]):
            best = (k, corr)
    
    return best

Trong scenario chính, đo payment-svc.latency_p99 và checkout-svc.latency_p99. Lag tốt nhất sẽ là k ≈ 5 (giây) — payment leading checkout by 5s. → payment có vẻ cause checkout.

3.4 PC algorithm — full causal graph#

PC algorithm (Peter-Clark) build causal DAG từ data:

  1. Start với fully connected undirected graph
  2. Remove edge nếu conditional independence test PASS
  3. Orient remaining edges theo collider patterns

Có library: causal-learn, pgmpy. Cho production AIOps đây là research direction, không phải go-to. Nhắc để biết.

3.5 Khi nào dùng causal vs graph#

Tình huốngTool tốt
Có service graph trust đượcGraph traversal — fast, deterministic
Service graph không đầy đủ, thiếu edgeCausal — học edge từ data
Đã trong middle of incidentGraph + temporal (Section 2.4) — < 1s
Post-mortem deep diveCausal + manual review

Cho lab này: dùng graph traversal + temporal chính, có thể đề cập causal trong reflection.

4. LLM-Augmented RCA#

Đến đây bạn có top-3 root cause candidate. LLM giúp:

  1. Diagnose loại root cause (“connection_pool_exhaustion” / “slow_query” / “rebalance_storm”)
  2. Suggest action dựa trên incident history
  3. Confidence reasoning — giải thích vì sao service X là root cause

4.1 Pipeline tổng#

cluster + top-3 candidate from graph RCA
        ↓
   build context:
     - cluster metadata
     - service criticality + team
     - similar past incidents (retrieve from history)
        ↓
   LLM prompt
        ↓
   structured output: {root_cause, class, confidence, actions, similar_incidents}

4.2 Retrieval — similar past incidents#

incidents_history.json có 30 incident lịch sử. Cần pick top-K incident similar nhất với cluster hiện tại.

Approach đơn giản — keyword similarity:

def incident_similarity(cluster: dict, history_item: dict) -> float:
    """
    Score 0-1: cluster hiện tại giống incident lịch sử bao nhiêu.
    
    Heuristic:
      +0.4 nếu cùng root_cause_service (or service trong cluster.services match history root_cause_service)
      +0.2 mỗi service trong cluster.services xuất hiện trong history.services_involved (max +0.4)
      +0.2 nếu severity match
    """
    score = 0.0
    
    if history_item['root_cause_service'] in cluster['services']:
        score += 0.4
    
    overlap = set(cluster['services']) & set(history_item['services_involved'])
    score += min(0.4, 0.2 * len(overlap))
    
    if cluster.get('max_severity') == history_item.get('severity'):
        score += 0.2
    
    return min(score, 1.0)


def top_k_similar(cluster: dict, history: list[dict], k: int = 3) -> list[dict]:
    scored = [(item, incident_similarity(cluster, item)) for item in history]
    scored.sort(key=lambda x: -x[1])
    return [{**item, '_similarity': s} for item, s in scored[:k] if s > 0.2]

4.4 Calling the LLM#

import os
import json

def call_llm_rca(cluster: dict, candidates: list, history: list[dict], graph_summary: str) -> dict:
    """
    Wrap LLM call. Trả về dict đã parse JSON.
    
    Note: log toàn bộ prompt + response vào file để evidence + debug.
    """
    similar = top_k_similar(cluster, history, k=3)
    
    similar_block = '\n'.join(
        f"- {i['id']}: root_cause={i['root_cause_service']} ({i['root_cause_class']}) | "
        f"{i['summary']} | remediation: {i['remediation']}"
        for i in similar
    ) or "(no similar incidents found)"
    
    ranked_block = '\n'.join(
        f"  {idx+1}. {svc}  (score: {score:.2f})"
        for idx, (svc, score) in enumerate(candidates[:5])
    )
    
    prompt = RCA_PROMPT_TEMPLATE.format(
        cluster_id=cluster['cluster_id'],
        time_range=' to '.join(cluster['time_range']),
        services=', '.join(cluster['services']),
        ranked_candidates=ranked_block,
        similar_incidents_block=similar_block,
        topology_block=graph_summary,
    )
    
    # OpenAI client (or Anthropic, Bedrock — interchangeable)
    from openai import OpenAI
    client = OpenAI(api_key=os.environ['OPENAI_API_KEY'])
    
    response = client.chat.completions.create(
        model='gpt-4o-mini',
        messages=[
            {'role': 'system', 'content': 'You are a senior SRE. Respond only in valid JSON.'},
            {'role': 'user', 'content': prompt},
        ],
        temperature=0.2,           # Low — RCA cần consistent, không creative
        response_format={'type': 'json_object'},
    )
    
    raw = response.choices[0].message.content
    
    # Log evidence
    with open('rca_llm_trace.log', 'a') as f:
        f.write(f"\n--- {cluster['cluster_id']} ---\n")
        f.write(f"PROMPT:\n{prompt}\n")
        f.write(f"RESPONSE:\n{raw}\n")
    
    return json.loads(raw)

4.5 Hallucination guard#

LLM có thể trả root_cause = service không có trong cluster. Validate trước khi dùng:

def validate_llm_output(parsed: dict, cluster: dict, allowed_classes: set) -> tuple[bool, list[str]]:
    """Return (is_valid, errors). errors list cụ thể vấn đề."""
    errors = []
    
    if parsed.get('root_cause') not in cluster['services']:
        errors.append(f"root_cause '{parsed.get('root_cause')}' not in cluster services")
    
    if parsed.get('class') not in allowed_classes:
        errors.append(f"class '{parsed.get('class')}' not in allowed enum")
    
    conf = parsed.get('confidence', 0)
    if not isinstance(conf, (int, float)) or not (0 <= conf <= 1):
        errors.append(f"confidence '{conf}' not in [0, 1]")
    
    actions = parsed.get('actions', [])
    if not isinstance(actions, list) or len(actions) == 0:
        errors.append("actions empty or wrong type")
    
    return (len(errors) == 0, errors)

Nếu invalid → fallback: dùng top-1 từ graph RCA + class=“other” + actions=[“Investigate manually”].

4.6 Cost awareness#

gpt-4o-mini (Jun 2026 price ≈ 0.15/1Minput,0.6 / 1M output):

  • Prompt ~2500 tokens × 1 call = $0.0004
  • Trong scenario, 1 batch = 3-5 cluster → 3-5 LLM call = ~$0.002
  • 1 năm production với 100 incident/ngày × 0.002=73/năm

Cheap. Nhưng nếu naive — gọi LLM cho mỗi alert thay vì mỗi cluster — sẽ tốn 50-100x.

5. Putting It Together — Final RCA Function#

def run_rca(cluster: dict, alerts: list[dict], graph: nx.DiGraph, history: list[dict]) -> dict:
    """
    Full RCA pipeline cho 1 cluster.
    
    Returns dict matching W2_lab_spec.md Section 4 schema (subset).
    """
    # Step 1: Graph-based ranking
    candidates = rca_combined(cluster, alerts, graph)  # [(svc, score), ...]
    
    if not candidates:
        return {'root_cause': None, 'confidence': 0, 'method': 'no-candidates'}
    
    # Step 2: Compose graph summary for LLM
    sub = graph.subgraph(cluster['services'])
    edges_str = '\n'.join(f"  {u} → {v}" for u, v in sub.edges())
    graph_summary = f"Service dependencies in cluster:\n{edges_str}"
    
    # Step 3: LLM enrichment
    try:
        llm_out = call_llm_rca(cluster, candidates, history, graph_summary)
        valid, errors = validate_llm_output(llm_out, cluster, allowed_classes={
            'connection_pool_exhaustion', 'slow_query', 'memory_leak', 'rebalance_storm',
            'deadlock', 'network_partition', 'bad_deploy', 'config_push', 'tls_expiry',
            'ddos', 'other'
        })
        if not valid:
            # Fallback: use graph top-1
            return {
                'root_cause': candidates[0][0],
                'class': 'other',
                'confidence': candidates[0][1],
                'actions': ['Investigate manually'],
                'reasoning': f'LLM output invalid: {errors}',
                'method': 'graph-only-fallback'
            }
        
        # Augment with raw graph score
        llm_out['graph_score'] = candidates[0][1]
        llm_out['method'] = 'graph+llm'
        return llm_out
    
    except Exception as e:
        return {
            'root_cause': candidates[0][0],
            'class': 'other',
            'confidence': candidates[0][1] * 0.5,  # discount for fallback
            'actions': ['LLM unavailable — investigate manually'],
            'reasoning': f'LLM call failed: {e}',
            'method': 'graph-only-llm-failed'
        }

Cũng viết FINDINGS.md (rubric yêu cầu) ≥ 100 từ, gồm:

  • Đối với cluster chính, root cause là gì + lý do
  • Confidence của bạn — có dám deploy auto-remediation dựa trên output này không
  • 1 case mà bạn không chắc — vì sao

6.3 Steps#

  1. Tạo aiops-<your-name>/w2/d2/assignment.ipynb
  2. Import D1 output (cluster_summary.json)
  3. Build graph từ services.json
  4. Cho mỗi cluster: chạy rca_combined → có top-K candidates
  5. Chuẩn bị incidents_history.json → retrieve top-3 similar
  6. Call LLM (nếu có key) hoặc skip + dùng graph only
  7. Validate output schema
  8. Write rca_output.json + FINDINGS.md + SUBMIT.md

6.4 Acceptance#

  • Notebook chạy được, ≥ 3 cell có output
  • rca_output.json valid, có graph_top3 + root_cause + class
  • Code có ít nhất 1 trong: networkx, nx\., graph, dependency
  • Code có 1 indicator của: RAG, retriev, LLM, openai, anthropic, ChatCompletion, langchain, llama (bonus, không bắt buộc)
  • FINDINGS.md ≥ 100 từ với RCA analysis cụ thể

7. EOD Checkpoint#

Trong SUBMIT.md, trả lời:

  1. Phân biệt culprit vs victim trong cluster của bạn — nêu 1 service victim, 1 service culprit, lý do.
  2. PageRank của bạn output gì cho cluster chính? Liệt kê top-3 với score.
  3. Granger causality có dùng trong code của bạn không? Nếu không, giải thích vì sao (3 sentence).
  4. LLM của bạn có hallucinate trong test runs không? Nếu có, ví dụ; nếu không, làm cách nào bạn defend?
  5. Confidence từ pipeline là 0.6. Bạn deploy auto-rollback ngay hay đợi SRE confirm? Vì sao?

Câu 4 + 5 là câu production maturity — sẽ rất giúp ở vấn đáp Friday.

8. Sang ngày mai#

D3 đưa pipeline của bạn (D1 correlate + D2 RCA) thành API service:

  • FastAPI endpoint nhận alert batch, trả incident response
  • Latency budget — RCA call qua LLM không được làm endpoint > 60s
  • A/B + shadow deploy cho model serving
  • Monitoring chính bản thân hệ thống AIOps

Đến hết D3, bạn có toàn bộ skill cần cho lab Friday.
